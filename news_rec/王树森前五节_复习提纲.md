## 0. 总框架（先记这个）

推荐系统不是一个模型，而是一个系统工程，典型链路是：

`召回 -> 排序 -> 重排 -> 上线实验（AB测试） -> 推全/回滚`

你刚听完的 5 节课在这个链路中的位置：

1. 基本概念：认识推荐系统到底在解决什么问题
2. 链路：理解分层架构（召回/排序/重排）
3. AB测试：理解线上如何验证方案有效
4. ItemCF：召回层经典方法（基于共现）
5. Swing：召回层改进方法（给共现去噪）

## 1. 概要01：推荐系统的基本概念

### 1.1 这节课核心目标

搞清楚推荐系统在干什么：

- 给“对的人”推荐“对的内容/商品/文章”
- 核心不是只做模型，而是优化业务目标（点击、时长、转化、GMV 等）

### 1.2 推荐系统常见输入（人话版）

1. 用户信息（User）
- 用户ID
- 历史行为（点击、收藏、停留时长）
- 上下文（设备、时间、来源）

2. 物品信息（Item）
- 商品/视频/新闻/文章ID
- 类别、发布时间、文本、图像等属性

3. 场景信息（Context）
- 当前时间
- 当前设备
- 当前页面/入口

### 1.3 推荐系统输出是什么

- 一串有顺序的候选列表（TopK）
- 顺序很重要（前面的位置价值更高）

### 1.4 推荐系统常见目标（先有概念）

- CTR（点击率）
- Watch Time / 停留时长
- Conversion（转化）
- GMV（交易额）
- 长期留存（更复杂）

### 1.5 你当前阶段要记住的一句话

推荐系统的目标不是“预测一个标签”这么简单，而是“在业务约束下生成高质量排序列表”。

## 2. 概要02：推荐系统的链路（召回 -> 排序 -> 重排）

### 2.1 为什么要分层

因为候选空间太大，不能直接对全量物品做重模型精排。

举例（新闻推荐）：
- 文章池有 36 万篇
- 不可能给每个用户每次都用复杂模型给 36 万篇全部打分

所以要先筛一小批，再精排。

### 2.2 三层职责（最重要）

1. 召回（Recall）
- 目标：从海量物品里快速捞出一小批候选（几十/几百/几千）
- 重点：速度快、覆盖广
- 方法：热门召回、ItemCF、Swing、双塔召回、Embedding相似召回等

2. 排序（Ranking）
- 目标：对召回候选做更准确的打分排序
- 重点：精度高（更细地估计点击/转化概率）
- 方法：LR、GBDT/LightGBM、Wide&Deep、DIN/DIEN 等

3. 重排（Re-ranking）
- 目标：在排序结果基础上做业务规则与体验优化
- 重点：多样性、新鲜度、打散、去重、业务约束

### 2.3 你现在和链路的关系（天池阶段一）

你当前重点在：

- 赛题理解（知道目标和评估）
- Baseline 跑通
- 召回层（ItemCF / Swing 思想）

还没到精排模型细节，不需要着急。

### 2.4 一句话记忆

召回负责“别漏掉”，排序负责“排得准”，重排负责“用得好”。

## 3. 概要03：推荐系统的 AB 测试

### 3.1 什么是 AB 测试（线上对照实验）

把用户随机分流成两组：

- A组：旧方案（对照组）
- B组：新方案（实验组）

比较指标差异（CTR、时长、转化等），判断新方案是否真的更好。

### 3.2 为什么需要 AB 测试

离线指标好，不一定线上就好。

原因可能有：
- 分布不一致
- 用户反馈环
- 延迟、工程实现差异
- 指标目标不一致

所以最终要看线上实验结果。

### 3.3 常见术语（你已经问过）

1. 实验推全
- 小流量实验效果稳定后，推到全量流量上线

2. 反转实验
- 对调实验桶再验证一次（看效果是否“跟着方案走”，不是“跟着桶走”）

3. 回滚
- 推全后若指标或系统异常，撤回新方案

### 3.4 哈希分流（你问过）

常见思路（稳定且“看起来随机”）：

```text
group = hash(user_id) % 100
0-49 -> A组
50-99 -> B组
```

好处：
- 同一个用户多次访问进同一组（稳定）
- 总体分布较均匀（便于对照）

### 3.5 常见坑（先知道名字）

- 分流不随机（组不公平）
- 样本不足（波动大）
- 多个改动一起上（无法归因）
- 指标看得太早（结论不稳）

### 3.6 一句话记忆

AB测试是推荐系统上线前的“对照实验”，不是主观感觉投票。

## 4. 召回01：ItemCF（基于物品的协同过滤）

### 4.1 核心直觉（人话）

如果很多用户都同时点了两篇文章 A 和 B，那么 A 和 B 很可能相似。

所以：
- 用户点过 A -> 可以给他推荐和 A 相似的 B

### 4.2 两个重要集合（一定要熟）

1. `U(i)`：点过物品 `i` 的用户集合
2. `I(u)`：用户 `u` 点过的物品集合

### 4.3 物品相似度（常见简化版）

当行为是 0/1（点过=1，没点过=0）时，常见余弦形式：

\[
sim(i,j)=\frac{|U(i)\cap U(j)|}{\sqrt{|U(i)|\cdot|U(j)|}}
\]

解释：
- 分子：共同用户数（共现程度）
- 分母：归一化（避免热门物品天然更占便宜）

### 4.4 给用户打召回分（核心思路）

对用户 `u` 的候选物品 `j`：

\[
score(u,j)=\sum_{i\in I(u)} w_{u,i}\cdot sim(i,j)
\]

- `I(u)`：用户历史点击过的物品
- `sim(i,j)`：历史物品和候选物品的相似度
- `w_{u,i}`：权重（可做时间衰减，先理解成 1 也行）

### 4.5 ItemCF 的优点（为什么它是 baseline 常客）

- 简单、直观、好解释
- 不需要复杂深度模型
- 在很多场景 baseline 很强

### 4.6 ItemCF 的典型问题（重点）

1. 热门物品偏置
- 热门物品和很多东西都有共同用户，容易“假相似”

2. 重度用户噪声
- 什么都点的用户会给很多物品对贡献共现，信息含量低

3. 只看共同用户数/共现，不够区分“共同用户质量”

这就是后面引出 Swing 的原因。

### 4.7 一句话记忆

ItemCF = 用“用户共现”来定义物品相似度，再按用户历史物品相似度召回新物品。

## 5. 召回02：Swing 模型

### 5.1 为什么会有 Swing（动机）

ItemCF 的问题是：
- 共同用户里如果有很多“非常像”的用户（比如重度用户），共现证据可能不够有辨识度

Swing 的改进思路：
- 不仅看共同用户数量
- 还看这些共同用户彼此像不像（用户对重合度）

### 5.2 核心直觉（必须记住）

共同用户彼此越像 -> 这条共现证据越“普通” -> 权重应更小  
共同用户彼此越不像 -> 这条共现证据越“稀缺” -> 权重应更大

### 5.3 常见形式（核心权重）

设 `V = U(i1) ∩ U(i2)` 为同时点过 `i1` 与 `i2` 的用户集合，用户对重合度：

\[
overlap(u_1,u_2)=|I(u_1)\cap I(u_2)|
\]

常见权重项：

\[
\frac{1}{\alpha + overlap(u_1,u_2)}
\]

- `alpha`：平滑项（避免分母过小，也控制惩罚强度）

对应相似度（示意）：

\[
sim(i_1,i_2)=\sum_{u_1,u_2\in V, u_1\neq u_2} \frac{1}{\alpha+overlap(u_1,u_2)}
\]

### 5.4 Swing 相比 ItemCF 的优势

- 更抗重度用户噪声
- 对“跨圈层共同点击”更敏感
- 在一些场景里相似度更有辨识度

### 5.5 Swing 的代价

- 计算更复杂（要看用户对）
- 工程实现比 ItemCF 更重

### 5.6 一句话记忆

Swing = 在 ItemCF 的“共同用户共现”基础上，再按“共同用户彼此重合度”做去噪加权。

## 6. ItemCF 与 Swing 对比（复习时重点看）

### 6.1 相同点

- 都是召回层方法
- 都利用用户历史行为（协同过滤）
- 都在做“物品相似度”

### 6.2 不同点

1. ItemCF
- 重点看“共同用户有多少”

2. Swing
- 重点看“共同用户有多少 + 这些共同用户彼此像不像”

### 6.3 你当前阶段该怎么用

- 先跑通 ItemCF baseline（阶段一）
- 用 Swing 建立“为什么要改进共现”的理解（先懂思想）
- 不要求立刻手写 Swing 工程版

## 9. 常见易混点（你复习时重点盯）

1. “推荐系统” != “一个模型”
- 它是系统链路 + 工程 + 实验验证

2. “AB测试好” != “一定立刻推全”
- 还要看稳定性、监控、风险、回滚方案

3. ItemCF 和 Swing 都是“召回”
- 不是排序模型

4. Swing 不是否定 ItemCF
- 是在共现基础上进一步去噪增强

5. 赛题理解不是走过场
- 它决定后面你看 baseline、做特征、做模型时的方向感
